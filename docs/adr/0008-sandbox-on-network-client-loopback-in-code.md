# Keep the App Sandbox on; grant only outgoing-network, enforce loopback in code

The Penna app target has been sandboxed (`ENABLE_APP_SANDBOX = YES`) since it was first scaffolded (#9) — it came from Xcode's default macOS app template, not a deliberate decision. A sandboxed app with no network entitlement cannot open *any* socket, **including loopback to `localhost`**, so every call to Ollama (`localhost:11434`, ADR-0004) was blocked before it left the app — surfacing as a misleading `-1003 "hostname could not be found"` because the sandbox stops the connection before DNS resolution (issue #22).

**Decision:** keep the sandbox **on** and grant the single minimal capability the app needs — outgoing client connections — via the build setting `ENABLE_OUTGOING_NETWORK_CONNECTIONS = YES` (which Xcode compiles to the `com.apple.security.network.client` entitlement). We use the build-setting form to match the project's existing style (`ENABLE_APP_SANDBOX`, `ENABLE_USER_SELECTED_FILES = readonly`) rather than introducing a separate `.entitlements` file and hand-wiring `CODE_SIGN_ENTITLEMENTS`. This does **not** weaken the sandbox: it stays on, and only this one outgoing-network key is added (verified by inspecting the built app's embedded entitlements).

**The privacy guarantee lives in code, not the sandbox.** macOS network entitlements are all-or-nothing — there is no host-level granularity, no "localhost-only" entitlement. So `com.apple.security.network.client` technically permits connections to *any* host, and the sandbox therefore **cannot** enforce ADR-0001's hard constraint that no personal text leaves the machine. That guarantee is enforced in `OllamaKit` instead: `OllamaClient` refuses any non-loopback `baseURL`, throwing `OllamaError.nonLoopbackHost` *before* the request reaches the network in both `generate()` and `generateStream()` (`isLoopbackHost` allows only `localhost`, `::1`, and `127.0.0.0/8`). This is covered by `LoopbackOnlyTests` — a regression test that pins the privacy property as a property of the code.

This bug stayed invisible to our checks because the two things that "worked" both bypass the sandbox: `swift test` injects a fake transport (never touches the network), and `ollama-tracer` is a command-line tool (not sandboxed). Neither exercises the packaged, sandboxed app's real network path — *the thing tested wasn't the thing that ships.* The loopback guardrail test now closes that gap at the code level; the only remaining check that requires a human is running the built app and observing a live Ollama response.
